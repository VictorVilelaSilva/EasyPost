Abordagens para a Refatoração

3 formas de estruturar o novo sistema de orçamentos

A

Server Actions + App Router (Recomendado)

Usa Server Components do Next.js 16 para leitura de dados e Server Actions para mutações. API routes só para PDF. Menos código cliente, mais rápido, sem estado global.

Prós

  • Menos boilerplate
  • Sem fetch manual na maioria das rotas
  • Formulários com action= nativo

Contras

  • Padrão mais novo, menos exemplos
  • Mudança em relação ao código atual
B

API Routes + Client Components

Mantém o padrão atual do EasyPost: API routes em /api/*, client components que fazem fetch. Familiar, previsível, mais fácil de debugar.

Prós

  • Padrão já usado no projeto
  • Fácil de testar APIs separado
  • Mais controle sobre loading states

Contras

  • Mais arquivos (route.ts pra tudo)
  • Fetch manual + loading/error em todo lugar
C

Híbrido: Server Components + API só pra PDF

Pages e listas como Server Components (leitura direta do Prisma). Formulários como Client Components com Server Actions. API route só para geração do PDF.

Prós

  • Melhor dos dois mundos
  • Performance boa sem complexidade extra
  • Fácil de evoluir

Contras

  • Mistura dois padrões
  • Requer entender Server vs Client components

Minha recomendação

Opção C (Híbrido) — aproveita o melhor do Next.js 16 sem abandonar Client Components onde fazem sentido (calculadora interativa, editor de orçamento). Simples de entender e manter.
